Purpose
Define the handover process and requirements between:
The team that defines data provisioning criteria (collection “initiators”/business owners)
The team that sets up policy and executes data provisioning (DPO/provisioning operations)
Focus covered both tactical (near-term) and future-state scenarios.
Main decisions/suggestions
Provisioning team wants early visibility of proposed collections to provide BAU perspective, challenge criteria, and flag user-group realities.
The criteria must be clear enough that they can be turned into specific metadata concepts. No ambiguous terms (e.g., “yellow studies”).
For the current Ranger-based provisioning process, a concrete list of study IDs/d-codes is mandatory. Criteria alone are insufficient.
Avoid broad org-driven lists (e.g., “all statisticians in oncology”). Instead, use specific tagged teams that someone actively manages to control who has access.
Until future system exists, an Excel-based tracker should/will be used.
All collection decisions and changes must be captured in a system-of-record. Tactically, it is suggested to have a timestamped email + Excel tracker. Future vision is to have a metadata-driven system with version history.
Define exactly which data modalities and from which source systems to provision, and where consumption is enabled (Domino/SCP/AI Bench/etc.). Boundaries of provisioning vs. consumption arrangements need a leadership decision.
Use a collections visibility “tracker” to coordinate BAU ad-hoc requests vs. upcoming collections; advise requesters to wait when sensible.
Roles in the process
Data Consumer Lead. A person who defines intent, proposes criteria, supplies the list of assets, nominates a Collection Leader.
Data Owner/Delegate Data Owner. A person who maintains the virtual team list (tags), validates collection criteria, monitors changes in user list, and communicates updates.
Provisioning Operations (DPO). A team which validates due diligence completion or executes pending checks, configures policies (Ranger/Starburst), executes provisioning across modalities and environments, updates trackers.
Systems Teams (Ranger/Immuta/Starburst/PDP/SCP/Domino/AI Bench/IO platform). Groups/Team/Platform Owners who execute changes as requested, support audit trails.
Data Office leadership/Value Stream 2. A group or a individual who decides default provisioning boundaries and supported consumption environments. They should set standard metadata model and future system adoption.

Process Steps
Business demand for a new collection or update to an existing one, or policy change (e.g., AI model training).
Early notification to DPO with initial intent, draft criteria, and anticipated users.
Translate draft criteria to meaningful (and static) metadata attributes.
Produce definitive asset list (study d-codes).
Identify and assign Data Consumer Lead, Data Owner
Obtain approved Data Collection artefacts (Open Access Collection, Agreement of Terms, trainings)
Confirm ethical/legal/data-owner approvals. If pending, assign actions and hold provisioning until cleared.
Tactical approach is to use email with attachments that includes Excel list of studies, Excel list of users (virtual team tags), modalities per study, source systems per modality, consumption environments, due diligence confirmations. Register the collection in iDAP for traceability (ticket number).
DPO configures Ranger/Starburst policies study-by-study. Executes data provisioning per modality and source.
Enable access in specified analysis environments (as per boundary decision).
Update tracker with per-modality, per-source, per-environment statuses.
Use a collections visibility tracker (summary) to inform the requesters about the status.
For conflicting requests (same data, different intents), cross-check tracker to avoid duplicates.
When criteria/users/assets change, update version and notify impacted teams.
If data owners revoke assets, auto-update policy where possible and notify the Data Collection Leader to cascade access revoke in other environments (e.g., AI Bench, SCP-EWF).
Map and track all locations where access was enabled to ensure complete and consistent access revoke across systems.
Maintain audit trail. Who approved what, when, and where provisioning occurred.
Record metrics (assets provisioned, users enabled, modalities and sources).
